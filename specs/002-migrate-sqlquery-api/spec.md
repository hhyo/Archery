# Feature Specification: SQLQuery Interface Unification

**Feature Branch**: `002-migrate-sqlquery-drf`  
**Created**: 2026-05-29  
**Status**: Draft  
**Input**: User description: "我希望将 sqlquery, 也就是查询SQL涉及到的所有接口, 改成 django rest framework 实现, 以充分利用到他的一些授权配置"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 安全执行查询 (Priority: P1)

作为有查询权限的用户，我希望继续在现有 SQL 查询页面上选择实例、数据库并执行查询，这些请求统一经过标准接口授权与校验，以确保查询结果、权限判断和安全限制保持一致。

**Why this priority**: 查询执行是该页面的核心价值，也是权限风险最高的能力；如果这一流程不能先稳定迁移，其余能力没有实际意义。

**Independent Test**: 仅实现查询执行相关接口迁移后，用户仍应能完成一次完整的实例选择、SQL 提交、结果查看流程，并正确触发权限校验、行数限制、超时控制、脱敏和审计记录。

**Acceptance Scenarios**:

1. **Given** 用户已登录且拥有目标实例的查询权限，**When** 用户在 SQL 查询页面执行一条允许的查询语句，**Then** 系统通过统一接口返回查询结果，并保留既有的限制行数、耗时、脱敏和主从延迟展示能力。
2. **Given** 用户没有目标实例或目标数据范围的查询权限，**When** 用户提交查询请求，**Then** 系统拒绝请求并返回明确的权限或范围错误，而不会泄露目标资源详情。
3. **Given** 用户提交了不符合查询规则的语句，**When** 系统进行服务端校验，**Then** 系统阻止执行并返回清晰的失败原因。

---

### User Story 2 - 管理历史与收藏 (Priority: P2)

作为日常使用 SQL 查询页面的用户，我希望查询历史、收藏、快捷重查等能力继续可用，并且这些能力遵循统一的可见性与授权规则，避免出现页面能看见但接口不允许，或接口暴露了不该看到的数据。

**Why this priority**: 历史与收藏直接影响高频操作效率，也是迁移后最容易出现授权不一致和数据泄露风险的区域。

**Independent Test**: 仅实现历史与收藏相关接口迁移后，用户应能筛选历史、收藏或取消收藏、从收藏发起重查，并且只能访问自己有权查看的记录。

**Acceptance Scenarios**:

1. **Given** 用户已有历史查询记录，**When** 用户按收藏状态、别名或关键字筛选查询历史，**Then** 系统返回分页结果，且结果范围符合用户角色可见性规则。
2. **Given** 用户对某条查询记录具有操作权限，**When** 用户收藏、取消收藏或维护别名，**Then** 系统保存变更并在后续列表与快捷选择中反映最新状态。
3. **Given** 用户尝试访问无权查看或修改的历史记录，**When** 系统接收该请求，**Then** 系统拒绝操作并保持原始记录不变。

---

### User Story 3 - 加载实例与表元数据 (Priority: P3)

作为使用 SQL 查询页面的用户，我希望实例列表、database 列表和 table 列表继续通过统一接口提供，这样页面上的实例、数据库和表选择器仍能正常联动。

**Why this priority**: 这些能力直接支撑查询主流程，但相对查询执行与历史管理更偏辅助，因此优先级略低。

**Independent Test**: 仅实现实例列表、database 列表和 table 列表接口迁移后，用户应能在页面中完成实例切换、database 加载和 table 加载的完整联动。

**Acceptance Scenarios**:

1. **Given** 用户进入 SQL 查询页面，**When** 页面加载其可访问实例列表，**Then** 系统仅返回该用户有权访问的实例。
2. **Given** 用户选择了一个可访问实例，**When** 页面请求该实例下的 database 列表，**Then** 系统返回该实例下用户可读取的 database 集合。
3. **Given** 用户已选择实例和 database，**When** 页面请求对应的 table 列表，**Then** 系统返回该范围内可用的 table 集合，并保持现有联动体验。

### Edge Cases

- 用户打开页面后权限发生变化，随后再次请求实例列表、database 列表、table 列表、查询或历史数据时，系统必须按最新权限立即拒绝无权操作。
- 查询执行成功但脱敏失败或超时控制触发时，系统必须返回一致的失败或降级结果，不得出现前端无法识别的异常格式。
- 用户使用相同页面流程切换不同数据库类型时，系统必须保证实例、database 和 table 加载结果始终对应当前选择，而不是沿用上一种数据库的上下文。
- 目标实例不存在、目标 database 不存在或目标范围下没有 table 时，系统必须给出清晰反馈，而不是返回与当前选择不一致的数据。
- 审计角色、普通用户和超级用户访问历史记录时，系统必须只暴露各自允许范围内的数据。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 将 SQL 查询页面当前范围内的 6 个后端交互能力迁移到统一的接口层，包括获取用户可访问实例、获取实例下的 database、获取实例与 database 下的 table、执行查询、读取历史查询记录、收藏或取消收藏 SQL 语句。
- **FR-002**: 系统 MUST 让迁移后的接口统一遵循现有集中式授权配置，确保认证方式、权限校验和资源范围控制在整组接口上表现一致。
- **FR-003**: 用户 MUST 能够继续通过现有 SQL 查询页面完成实例选择、数据库选择、语句提交和结果查看，而无需新增额外学习成本或手工步骤。
- **FR-004**: 系统 MUST 在查询执行接口中保留现有服务端安全控制，包括实例访问校验、语句合法性校验、查询权限校验、返回行数限制、超时终止、结果脱敏和审计留痕。
- **FR-005**: 系统 MUST 在查询历史接口中支持分页、筛选和关键字搜索，并按用户角色限制可见范围，避免返回越权的历史记录。
- **FR-006**: 系统 MUST 允许用户对有权访问的查询记录执行收藏、取消收藏和别名维护，并阻止对无权记录的修改。
- **FR-007**: 系统 MUST 提供用户可访问实例列表读取能力，并确保返回结果与当前用户的资源组和授权范围一致。
- **FR-008**: 系统 MUST 提供实例下 database 列表读取能力，并在返回前执行实例访问范围校验。
- **FR-009**: 系统 MUST 提供实例与 database 下 table 列表读取能力，并确保返回结果与当前实例和 database 选择一致。
- **FR-010**: 系统 MUST 为迁移后的各接口返回稳定、一致、可预期的成功与失败响应，使现有页面能正确处理加载、错误提示和结果展示。
- **FR-011**: 系统 MUST 为查询页面依赖的所有已迁移接口建立清晰的接口清单与迁移边界，确保发布时不存在遗漏的旧入口依赖。
- **FR-012**: 系统 MUST 让审计角色、普通用户和超级用户在迁移后继续保持与现有业务规则一致的数据访问差异。

### Test Strategy Constraints *(mandatory)*

- **TSC-001**: 验证方案 MUST 优先覆盖接口授权、资源范围控制、查询保护、历史可见性和响应契约的 pytest 单元测试。
- **TSC-002**: 共享测试准备 MUST 通过 conftest.py 或可复用夹具统一管理，不允许为每个接口重复构建用户、实例、权限和查询日志数据。
- **TSC-003**: 集成测试 MUST 仅用于证明跨越认证、权限、视图和现有查询引擎边界的关键行为，例如统一授权是否真正生效。
- **TSC-004**: 任何新增集成测试 MUST 包含简短说明，解释为什么该行为无法仅通过单元测试充分证明。

### Key Entities *(include if feature involves data)*

- **查询执行请求**: 表示用户在 SQL 查询页面发起的一次查询动作，包含目标实例、目标数据库、可选资源范围和待执行语句，并受权限、限制行数和安全规则约束。
- **查询结果记录**: 表示一次查询完成后保存的结果摘要，包含查询语句、目标实例、目标数据库、执行耗时、返回行数、收藏状态和可见性范围。
- **实例与资源列表请求**: 表示页面为联动选择器发起的资源读取动作，包括用户可访问实例、实例下的 database，以及实例与 database 下的 table。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: SQL 查询页面识别出的全部后端依赖能力均完成迁移映射，并在验收清单中达到 100% 覆盖，无遗留必经旧接口。
- **SC-002**: 在回归验证中，授权用户完成“选择实例并执行查询”主流程的操作步骤数不高于当前流程，且 95% 以上样例可一次完成。
- **SC-003**: 在验收测试中，针对未授权用户、越权资源和越权历史记录的访问样例，100% 被统一授权策略正确拒绝。
- **SC-004**: 与 SQL 查询页面直接相关的既有回归场景中，95% 以上在迁移后保持原有业务结果与页面反馈一致。

## Assumptions

- 本次特性仅覆盖已明确的 6 个接口，页面本身继续保留现有入口与交互方式，只进行适配迁移所需的最小前端改动。
- 现有查询引擎、脱敏逻辑、权限规则、审计记录和 AI 生成能力继续复用，本次不重新设计这些业务规则。
- 统一授权配置已经在现有 API 体系中可用，本次目标是让 SQL 查询相关接口接入并遵循这套配置，而不是新增一套独立授权模型。
- 表结构查看、AI 生成 SQL、线下导出工单和查询权限申请等相邻流程不在本次核心迁移范围内，除非后续重新扩展范围。
